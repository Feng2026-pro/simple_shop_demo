# 场景评审 Excel 固定列模板（精简版）

此模板用于“人读优先”的业务场景评审，统一承载**单接口用例 + 多接口串联场景**。  
主输入推荐使用 **场景 JSON**（根键 `cases`），由 `scenario_json_to_excel.py` 生成对应 `.xlsx`。

## 固定列顺序

| 序号 | 列名 | 必填 | 说明 |
|---|---|---|---|
| 1 | 场景类型 | 是 | `单接口` 或 `串联` |
| 2 | 场景ID | 是 | 如 `ATOM_001` / `SCN_CHAIN_001`，全表唯一 |
| 3 | 场景名称 | 是 | 一句话描述场景 |
| 4 | 测试点 | 是 | 本场景要覆盖的核心业务点 |
| 5 | 优先级 | 否 | 建议 `P0/P1/P2`，默认 `P1` |
| 6 | 前置条件 | 否 | 账号、基础数据、环境要求 |
| 7 | 前置依赖 | 否 | 依赖名，如 `dep_login_admin` |
| 8 | 依赖产出提取 | 否 | 提取说明，如 `token=$.data.token` |
| 9 | 注入方式 | 否 | 注入说明，如 `Authorization: Bearer DEP:dep_login_admin.token` |
| 10 | 步骤 | 是 | 单接口可写 1~2 步；串联建议编号多行 |
| 11 | 预期结果 | 是 | 业务结果与关键校验点 |
| 12 | 备注 | 否 | 边界说明、风险提示等 |

## 填写规范

- `步骤` 建议使用编号格式，明确接口调用顺序与参数来源。
- 对“先调 A 取值，再调 B/C”的场景，要在步骤中写清参数传递关系。
- 若步骤中存在可复用登录，建议沉淀为顶层 `dependencies`，并在本表通过 `前置依赖/注入方式` 表达，不在每条场景重复写登录动作。
- 不要求在本表中写可执行 JSON 结构；以可读性和可评审性优先。

## 覆盖充分性验收项

- 按模块统计 `场景类型=单接口` 与 `场景类型=串联` 的数量。
- 每个模块应出现更新、查询、删除相关场景，不可只保留创建或主链路。
- 写操作（POST/PUT/DELETE）至少包含 1 条负向用例（参数非法、状态非法、业务冲突或资源不存在）。
- 含数值字段的接口至少包含 1 条边界用例（0/负数/最小合法值）。

## 通用覆盖模板（跨项目）

- 查询类（Read）：列表成功、详情存在成功、详情不存在失败；若有权限模型再补越权负向。
- 创建类（Create）：创建成功、必填缺失、类型/枚举非法、重复创建冲突。
- 更新类（Update）：更新成功、非法状态流转/字段非法、资源不存在。
- 删除类（Delete）：可删成功、不可删约束（被引用/状态限制/业务冲突）、重复删除行为。
- 边界类（Boundary）：数值字段至少覆盖 `0`、负数、最小合法值（可判断时再补超限）。
