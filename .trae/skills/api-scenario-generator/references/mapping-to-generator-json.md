# 场景 Excel -> 执行 JSON 映射规则

**入口约定**：执行 JSON **始终**由场景评审 Excel 经本规则映射得到，避免与「场景 JSON」双套映射。场景 Excel 可由 `scenario_json_to_excel.py` 从 `*_scenario.json` 生成（见 [scenario-json-schema.md](scenario-json-schema.md)），再执行本流程。

目标：将场景评审 Excel 转成可供 `api-test-generator` / runner 使用的 JSON 用例。

## 输出 JSON 结构

支持两种根结构，推荐：

```json
{
  "cases": [
    {
      "用例ID": "SCN_001",
      "用例名称": "登录成功",
      "接口路径": "/auth/login",
      "路径参数": "",
      "请求方法": "POST",
      "请求体/参数": "{\"username\":\"user001\",\"password\":\"pass123\"}",
      "预期状态码": 200,
      "预期响应校验": "$.code==200 && $.data.token",
      "优先级": "P0",
      "前置依赖": "无",
      "依赖产出提取": "",
      "注入方式": "",
      "是否运行": "是",
      "运行结果": ""
    }
  ]
}
```

## 字段映射

| 场景 Excel 列 | 执行 JSON 字段 | 规则 |
|---|---|---|
| 场景ID | 用例ID | 原样 |
| 场景名称 | 用例名称 | 原样 |
| 接口路径 | 接口路径 | 原样 |
| 请求方法 | 请求方法 | 统一转大写 |
| 风险等级 | 优先级 | 原样（P0/P1/P2） |
| 是否转执行用例 | 是否运行 | `Y -> 是`，其他 -> `否` |
| 关键输入（自然语言） | 请求体/参数 | 仅当 `JSON:{...}` 可解析时写入；否则进入 issues |
| 预期结果（自然语言） | 预期响应校验 | 仅当 `CHECK:...` 可解析时写入；否则进入 issues |
| 预期状态码 / 预期状态码（HTTP） | 预期状态码 | 缺省 `200`；须为整数 |
| 路径参数 | 路径参数 | 须为合法 JSON 对象字符串；空为 `""` |
| 前置依赖 | 前置依赖 | 缺省 `无` |
| 依赖产出提取 | 依赖产出提取 | 原样写入 JSON（runner 对 YAML 依赖主要用 yaml 内 extract） |
| 注入方式 | 注入方式 | 原样写入 JSON |

## 固定默认值

| 执行 JSON 字段 | 默认值 |
|---|---|
| 路径参数 | `""` |
| 预期状态码 | `200`（场景表填写时以场景列为准） |
| 前置依赖 | `无` |
| 依赖产出提取 | `""` |
| 注入方式 | `""` |
| 运行结果 | `""` |

## 失败处理（关键）

若出现以下情况，不生成该条执行 JSON，并写入 `mapping_issues.md`：

1. `是否转执行用例 != Y`
2. 缺少必填列（如 场景ID/接口路径/请求方法）
3. `关键输入（自然语言）` 不能解析为 `JSON:{...}`
4. `预期结果（自然语言）` 不能解析为 `CHECK:...`
5. 请求方法不在允许集合 `GET/POST/PUT/DELETE`

issues 记录格式建议：

```markdown
- SCN_003 登录失败场景：关键输入无法结构化，请改为 JSON:{...}
- SCN_007 查询详情：预期结果无法结构化，请改为 CHECK:...
```

## 与现有规范对齐

- `预期响应校验` 语法对齐 `api-test-generator`（`==/!=/>/>=/</<=`、`&&`）。
- 生成 JSON 后可直接调用 `api-test-generator/scripts/doc_to_excel.py` 转执行 Excel。
